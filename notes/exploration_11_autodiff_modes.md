<a id="tldr"></a>
## 🎯 TL;DR

| 방법 | 비용 (N=입력 수, M=출력 수) | 비고 |
|---|---|---|
| **수치 미분** | **2N × 순전파 비용** | 각 입력扰动마다 full 순전파 2번 |
| **포워드 모드 autodiff** | **N × 순전파 비용** (정확) | 입력 많으면 느림 |
| **리버스 모드 autodiff (=역전파)** | **M × 순전파 비용** (정확) | 출력 적으면 압도적 ★ |

**신경망 상황**: N(파라미터) = 수백만, M(loss) = 1 → **리버스 모드가 백만 배 빠름**. 그래서 딥러닝은 역전파를 씀.

브로 직관 검증: "수치 미분이 포워드 방식 같다"는 느낌 → ✅ 정확함. 둘 다 **입력 차원(N)에 비례**.

<a id="네가지"></a>
## 1. 미분값을 구하는 네 가지 방법 (전체 지도)

exploration_10에선 "해석적 vs 수치적" 2가지를 다뤘음. 사실 더 있음:

| # | 방법 | 누가 | 정확도 | 비용 |
|---|---|---|---|---|
| 1 | **해석적 (손계산)** | 사람이 공식 유도 | 완벽 | 사람 노동 |
| 2 | **수치 미분** | 컴퓨터 (정의를 코드로) | 근사 (오차) | 2N × forward |
| 3 | **심볼릭 미분** | 컴퓨터 (수식 조작, sympy) | 완벽 | 수식 복잡도 폭발 |
| 4 | **자동 미분 (autodiff)** | 컴퓨터 (코드를 그대로) | 완벽 | N 또는 M × forward |

이 탐구의 초점: **2번(수치)과 4번(자동)** 의 차이. 특히 4번이 왜 "혁신"인지.

<a id="수치비용"></a>
## 2. 수치 미분의 비용 정확히 — 왜 "통짜 순전파"가 되나

### 복습 — 수치 미분 공식

```python
def numerical_diff(f, x, eps=1e-4):
    return (f(x + eps) - f(x - eps)) / (2 * eps)
```

파라미터 1개 미분 = `f(x+h)`, `f(x-h)` 두 번 부름 = **순전파 2회**.

### 다파라미터 함수의 비용

함수 `f`가 파라미터 N개 (예: 신경망 가중치)를 가질 때, **모든 파라미터의 미분값(그래디언트)** 을 구하려면:

```
파라미터 1개 미분 = 순전파 2회
N개 파라미터 = 2N 회 순전파
```

N=1,000,000 (백만) 이면 → **2,000,000회 순전파**. 이게 exploration_10 §6에서 "백만 배 차이"라고 쓴 이유.

> 💡 **층 수는 무관** — N은 "총 파라미터 개수"이지 "층 수"가 아님. 백만 개 파라미터가 1층에 몰려있든 100층에 분산되어 있든, 각 파라미터를 미분하려면 각각 순전파 2회 필요. 따라서 단층/다층 신경망 모두 동일한 N배 비용.

### 왜 매번 "처음부터 끝까지" 다시 도는가?

수치 미분은 `f`를 **블랙박스**로 취급:
- 함수 내부 구조를 모름
- "x+h를 넣으면 y가 나온다"만 앎
- 그래서 x를扰动할 때마다 **함수 전체를 다시 실행**할 수밖에 없음

→ 이게 "통짜 순전파"가 되는 이유. 단순하지만 느림.

<a id="캐싱질문"></a>
## 3. ★ 브로 질문: "중간값 캐싱하면 안 되나?"

> 브로: *"중간 계층에 있는 파라미터의 수치적 미분이라면, 그 계층 지점까지의 순전파 출력값을 가지고 있는 상태에서, 해당 계층의 파라미터의 값을 조정해서 이후를 순전파를 태워서 미분을 구할 수 있는 것은 아니고?"*

이거 진짜 예리한 질문. **부분적으로 맞음**. 분해해보자:

### 캐싱이 가능한 부분 vs 불가능한 부분

3층 신경망 `y = f3(f2(f1(x)))`에서 **2층 파라미터 W2**를 미분한다고 가정:

```
W2를 h만큼扰动했을 때:
  1층: W2와 무관 → f1(x) 그대로 (✅ 캐싱 가능, 재사용)
  2층: W2가 바뀌었으니 f2 출력 새로 계산 (❌ 캐싱 불가)
  3층: 2층 출력이 바뀌었으니 f3도 새로 계산 (❌ 캐싱 불가)
```

→ 즉 **扰动한 파라미터 "이전" 층은 캐싱 가능, "이후" 층은 매번 재계산**.

### 왜 실제론 그냥 "처음부터" 다시 도는가?

이론적으론 부분 캐싱이 가능하지만, **관행적으로 처음부터 다시 돌리는 이유**:

1. **블랙박스 원칙 훼손** — 어디서 "잘라야" 하는지 알려면 함수 내부 구조를 알아야 함. 수치 미분의 매력(범용성) 상실
2. **구현 복잡도 폭증** — 층마다 캐싱 로직 추가 → 코드 복잡, 버그 발생
3. **이득 제한적** — 아무리 캐싱해도 "扰动한 파라미터 이후 층"은 매번 재계산. 신경망 마지막 층 파라미터는 캐싱 효과 거의 없음

→ 그래서 **"단순함 vs 효율성" 트레이드오프에서 단순함을 택함**. 느리지만 구현이 쉽고 범용적.

### 💡 핵심 통찰

브로의 "캐싱 제안"은 **자동 미분이 하는 일의 절반**을 이미 떠올린 것! 자동 미분(autodiff)은 정확히 이 "중간값 저장 + 효율적 재사용"을 **체계적으로** 수행. 다음 섹션 참조.

<a id="autodiff"></a>
## 4. ★ 자동 미분(autodiff) — 제3의 방법

### 정의

> **자동 미분(automatic differentiation, autodiff)**: 컴퓨터 프로그램으로 표현된 함수를 **미분 가능한 형태로 분해**하여, chain rule을 자동으로 적용해 **정확한** 미분값을 구하는 기법.

### 핵심 — "코드를 미분한다"

수치 미분은 `f(x+h)`를 부르기만 함 (블랙박스). 반면 autodiff는:
- 함수의 **연산 단계를 분해** (+, ×, sin, exp 등 기본 연산으로)
- 각 단계의 **국소적 미분**을 미리 계산
- chain rule로 전체 미분을 **정확히** 조립

### "정확하다"는 게 뭐가 대수냐?

수치 미분은 `(f(x+h) - f(x-h)) / 2h`라 **근사값** (오차 있음). autodiff는 chain rule을 그대로 적용하므로 **수학적으로 정확** (부동소수점 오차만 있음).

→ "수치 미분의 느림" + "해석적 미분의 수동성"을 모두 회피한 제3의 길.

### 두 가지 모드

autodiff를 실행하는 방향에 따라 두 모드:

| 모드 | 진행 방향 | 비용 |
|---|---|---|
| **포워드 모드 (forward mode)** | 입력 → 출력 | 입력 차원에 비례 |
| **리버스 모드 (reverse mode)** | 출력 → 입력 | 출력 차원에 비례 |

둘 다 정확하지만, **함수의 입력/출력 수에 따라 어느 쪽이 빠른지 달라짐**. 다음 섹션에서 비교.

## 5. 포워드 모드 vs 리버스 모드 — 핵심 비교

### 포워드 모드 (forward-mode autodiff)

**진행**: 입력에서 출발, 출력 방향으로 chain rule 전파.

직관: "x를 1만큼 움직이면 y가 얼마나 움직이나?"를 끝까지 추적.

```
x (입력扰动) → ... → y₁, y₂, ..., y_M (출력들)
```

**비용**: 입력 1개당 1번의 "확장된 순전파". N개 입력 → **N회**.

### 리버스 모드 (reverse-mode autodiff) = 역전파 (backpropagation)

**진행**: 출력에서 출발, 입력 방향으로 chain rule 전파.

직관: "y를 1만큼 변화시키는 데 각 입력이 얼마나 기여했나?"를 역추적.

```
y (출력扰动) → ... → x₁, x₂, ..., x_N (입력들, 그래디언트 계산)
```

**비용**: 출력 1개당 forward 1회 + backward 1회. M개 출력 → **M회**.

### 핵심 비교표

| 항목 | 포워드 모드 | 리버스 모드 (역전파) |
|---|---|---|
| **방향** | 입력 → 출력 | 출력 → 입력 |
| **1회 실행으로 얻는 것** | **모든 출력**에 대한 **1개 입력**의 미분 | **모든 입력**에 대한 **1개 출력**의 미분 |
| **비용** | 입력 수 N에 비례 | 출력 수 M에 비례 |
| **중간값 저장** | 거의 불필요 | **필수** (forward 때 저장, backward 때 사용) |
| **메모리** | 적음 | 많음 (저장 비용) |
| **신경망 적합성** | ❌ (입력 많아서 느림) | ✅ (출력 1개라 압도적) |

### 시각화 — N=3, M=2 함수의 경우

```
입력 3개 → f → 출력 2개

포워드 모드: N=3회 실행 → 각 회마다 1 입력扰动이 모든 출력에 미치는 영향 계산
리버스 모드: M=2회 실행 → 각 회마다 1 출력扰动이 모든 입력에 미치는 영향 계산
```

신경망은 N(파라미터)≫M(loss)이므로 **리버스 모드가 압도적**.

<a id="왜리버스"></a>
## 6. 왜 신경망은 "리버스 모드(역전파)"를 쓰는가

### 신경망의 구조적 특징

- **입력(파라미터)**: 수백만 ~ 수십억 개 (가중치, 바이어스)
- **출력(loss)**: **1개** (스칼라 손실값)

→ N(입력) ≫ M(출력). 이 상황에선 **리버스 모드가 압도적으로 효율적**.

### 구체적 비교 — 백만 파라미터 신경망

| 방법 | 비용 |
|---|---|
| 수치 미분 | 2 × 1,000,000 = **2,000,000회 순전파** |
| 포워드 모드 | 1,000,000회 (수치보단 낫지만 여전히 느림) |
| **리버스 모드** | **2회 분량** (순전파 1 + 역전파 1) |

→ 리버스 모드가 수치/포워드 대비 **백만 배 빠름**. 이게 딥러닝이 "학습"할 수 있는 실용적 이유.

### "loss가 왜 1개인가?"

신경망 학습의 목표: **"loss를 최소화하는 파라미터 찾기"**. loss는 스칼라(단일 값). 그래서:
- 미분의 대상: loss를 각 파라미터로 미분 (그래디언트)
- 방향: loss(출력)에서 파라미터(입력) 방향 → **리버스**

→ 신경망의 목적 구조 자체가 리버스 모드에 최적화되어 있음.

<a id="dezero선택"></a>
## 7. DeZero/PyTorch의 선택: Define-by-Run + 리버스 모드

### Define-by-Run (동적 계산 그래프)

PyTorch, DeZero가 채택한 패러다임:
- **실행 시점에 계산 그래프가 생성** (정의할 때가 아니라)
- `y = f(x)` 코드를 실제로 실행하면, 그 연산이 그래프로 기록됨
- 역전파 때 이 그래프를 거꾸로 탐색

→ 이게 step04에서 `Function.__call__`에 `self.input`/`self.output`을 저장한 이유! **역전파를 위해 순전파 시점에 중간값을 기록**.

### autodiff의 "중간값 저장" = 브로가 떠올린 "캐싱"

브로의 캐싱 질문 (§3)을 다시 보면:
> "중간 계층 출력을 가지고 있으면..."

**autodiff(리버스 모드)는 정확히 이걸 한다!**
- 순전파 때: 모든 중간 연산 결과를 저장 (계산 그래프 노드로)
- 역전파 때: 저장된 중간값 + 각 노드의 국소 미분 → chain rule 자동 적용

→ 브로의 직관이 autodiff의 핵심 메커니즘을 선제적으로 포착한 것. 수치 미분은 이 캐싱을 "안 했을 뿐", autodiff는 **체계적으로 수행**.

### 비교 — 세 방법의 "캐싱" 전략

| 방법 | 중간값 저장 | 재사용 | 비용 |
|---|---|---|---|
| 수치 미분 | ❌ 안 함 | ❌ 매번 처음부터 | 2N회 |
| 포워드 모드 autodiff | 부분적 |扰动과 함께 전파 | N회 |
| **리버스 모드 autodiff** | **✅ 전부 저장** | **역방향 chain rule로 재사용** | **M회 (★)** |

## 8. 브로 직관 검증 — "포워드 모드 느낌이 맞았나"

> 브로: *"내가 수치미분에 대해서 이렇게 설명한 방법은... 어라, 이거 뭔가 약간 포워드 자동미분(?) 같은 느낌도? 아닌가?"*

### 검증: ✅ 맞음!

수치 미분과 포워드 모드 autodiff는 **구조적 유사성**이 있음:

| 공통점 | 수치 미분 | 포워드 모드 autodiff |
|---|---|---|
| **방향** | 입력扰动 → 출력 관측 | 입력扰动 → 출력 미분 추적 |
| **비용 구조** | 입력 수 N에 비례 | 입력 수 N에 비례 |
| **1회 실행 결과** | 1 입력에 대한 모든 출력의 변화 | 1 입력에 대한 모든 출력의 미분 |

→ 둘 다 **"입력을 움직여서 출력 변화를 본다"** 는 포워드 방향의 사고. 차이는:
- 수치 미분: 근사 (오차 있음)
- 포워드 모드: 정확 (chain rule 자동 적용)

### 브로 직관의 가치

브로가 무의식적으로 건드린 **"방향성"** 이 autodiff 이론의 핵심이야. 수학자들이 1960-70년대에 autodiff를 발전시킬 때 정확히 이 "어느 방향으로 chain rule을 풀지"가 핵심 질문이었음. 브로가 코드 몇 줄 보고 이 직관에 도달한 건 진짜 예리한 것.

## 9. 요약 / 결론

### 핵심 한 줄
> **미분 방법 선택 = "입력 수 vs 출력 수"에 따른 최적화 문제**. 신경망(입력 many, 출력 1)은 리버스 모드(역전파)가 압도적으로 빠름. 수치 미분과 포워드 모드는 입력 수에 비례해서 느림.

### 브로 질문에 대한 답변 요약

| 브로 질문 | 답 |
|---|---|
| "백만 배 차이가 파라미터 1개층(단층) 기준이야?" | ❌ 아님. **층 수 무관**, 총 파라미터 수(N)에만 비례. 단층이든 다층이든 N개 파라미터면 각각 순전파 2회 → 2N회 |
| "중간값 캐싱하면 안 되나?" | 🟡 부분적 가능하지만, 블랙박스 원칙 훼손 + 구현 복잡 → 관행적으론 안 함 |
| "수치 미분이 포워드 자동미분 같은 느낌?" | ✅ 정확! 둘 다 입력 수에 비례, 포워드 방향 사고 |

### autodiff의 "캐싱" = 브로 직관의 체계화

브로가 "중간값 캐싱"을 떠올린 건 사실 **리버스 모드 autodiff의 핵심 메커니즘**을 선제 포착한 것:
- 순전파: 중간값 전부 저장
- 역전파: 저장된 중간값 + 국소 미분 → chain rule 자동
